* Risk Guardrails

This document records the non-negotiables for the optimization branch.

** Must not regress

1. Permission request rendering and auto-approval flow.
2. Tool call updates, including plan and diff handling.
3. User message chunk insertion and transcript formatting.
4. Viewport navigation, collapse/expand, and history restore.
5. Header/mode-line correctness after session/model/mode changes.

** Performance change rules

1. If a patch changes displayed content, add or update a functional test.
2. If a patch changes update frequency, capture before/after metrics.
3. If a patch changes caching, add invalidation coverage.
4. If a patch touches notification handling, compare with the baseline profile.
5. If a patch changes fragment reuse, add tests that prove `agent-shell-ui-update-fragment` still updates and re-reads the same block correctly.
6. If a patch caches mode/model menus, validate that menu labels change after session state updates.

** Known environment blocker

The current batch test runner can fail in this environment because the bundled Emacs binary depends on a mismatched `libmount.so.1`/`libgio-2.0.so.0` combination when started with the default dynamic loader state.

We can still use the same Emacs binary with a scrubbed environment, but loading runtime dependencies (`acp`, `shell-maker`) must be made available before the test suite can execute end-to-end.

** Planning rule

If `.org/plans/optimization-roadmap.org` and the code diverge, update the plan first so the next patch is still constrained by the real state of the branch.

** Active change risk

The next patch will touch `agent-shell--on-notification`. Treat all branches in that function as user-visible unless a test proves otherwise.
